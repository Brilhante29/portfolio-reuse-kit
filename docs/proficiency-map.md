# Proficiency Map

This kit should help the portfolio prove the skills Guilherme wants to be known for on GitHub and LinkedIn.

The stack is not chosen only because it is popular. It is chosen when it helps prove the project problem and strengthens a public proficiency signal.

## Decision Order

1. Define the problem the repository solves.
2. Place it in a portfolio program.
3. Decide what public proficiency it should demonstrate.
4. Choose architecture from problem forces.
5. Choose stack/framework from the proficiency map and language profile.
6. Choose benchmark that turns the repo into a post.

## Frontend Signals

Angular should be used when the repo should prove enterprise frontend skill: feature modules, typed forms, route guards, service/facade boundaries, and dashboard/admin workflows.

Next.js should be used when the repo should prove React product frontend skill: public demo quality, SSR/ISR/rendering tradeoffs, route structure, polished dashboards, and shareable UI.

## Data and ML Signals

Python remains the default for data, ML, CV, RAG, and MLOps.

MLflow should appear when the project needs experiment tracking, model comparison, registry behavior, or lifecycle evidence.

Airflow should appear when the project needs orchestration: scheduled DAGs, dependencies, retries, backfills, and pipeline observability.

Notebook-style teaching is useful for ML, but it cannot be the only artifact. Every portfolio repo still needs scripts, tests, Docker, README result, and benchmark JSON.

## Paulescu Reference

Use Paulescu's ML style as inspiration for progressive, problem-by-problem learning and community-facing clarity. The useful pattern is not the exact stack; it is the organization: chapters by problem, clear README narrative, and hands-on progression.
